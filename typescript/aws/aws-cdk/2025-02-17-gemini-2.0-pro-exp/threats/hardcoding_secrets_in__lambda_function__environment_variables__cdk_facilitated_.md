# Deep Threat Analysis: Hardcoded Secrets in AWS CDK Lambda Functions

## 1. Objective

The objective of this deep analysis is to thoroughly understand the threat of hardcoding secrets within the `environment` property of the `lambda.Function` construct in AWS CDK applications.  We aim to identify the root causes, potential attack vectors, and the effectiveness of various mitigation strategies, providing actionable recommendations for development teams.  This analysis goes beyond a simple description and delves into the *why* and *how* of the threat within the CDK ecosystem.

## 2. Scope

This analysis focuses specifically on:

*   **AWS CDK:**  The threat as it manifests within applications built using the AWS Cloud Development Kit (CDK).  We are *not* analyzing general Lambda secret management outside the CDK context.
*   **`lambda.Function` Construct:**  The primary target is the `environment` property of the `aws-cdk-lib/aws-lambda` module's `lambda.Function` construct.
*   **Hardcoded Secrets:**  We are concerned with secrets (API keys, database credentials, etc.) directly embedded as string literals within the CDK code.
*   **Compromise of CDK Code Repository:** The primary attack vector assumes an attacker gaining access to the source code repository (either public or private) where the CDK code resides.

This analysis *excludes*:

*   Threats related to Lambda function code vulnerabilities (e.g., injection attacks).
*   Threats related to IAM role misconfigurations (unless directly related to secret retrieval).
*   Threats arising from compromised AWS credentials *outside* the context of the CDK code itself.

The analysis will follow these steps:

1.  **Threat Modeling Review:**  Reiterate the core threat and its impact, ensuring a clear understanding of the problem.

**Secure Code (Using Secrets Manager):**

```typescript
import * as cdk from 'aws-cdk-lib';
import * as lambda from 'aws-cdk-lib/aws-lambda';
import * as secretsmanager from 'aws-cdk-lib/aws-secretsmanager';

export class SecureStack extends cdk.Stack {
  constructor(scope: cdk.App, id: string, props?: cdk.StackProps) {
    super(scope, id, props);

    // Retrieve the secret from Secrets Manager
    const mySecret = secretsmanager.Secret.fromSecretNameV2(this, 'MySecret', 'my-secret-name');

    const myFunction = new lambda.Function(this, 'MyFunction', {
      runtime: lambda.Runtime.NODEJS_18_X,
      handler: 'index.handler',
      code: lambda.Code.fromAsset('lambda'),
      environment: {
        // Access the secret value at runtime
        DATABASE_PASSWORD: mySecret.secretValueFromJson('database_password').toString(),
        API_KEY: mySecret.secretValueFromJson('api_key').toString(),
      },
    });

    // Grant the Lambda function read access to the secret
    mySecret.grantRead(myFunction);
  }
}
```

```typescript
// lib/my-stack.ts
import * as cdk from 'aws-cdk-lib';
import * as lambda from 'aws-cdk-lib/aws-lambda';
import * as secretsmanager from 'aws-cdk-lib/aws-secretsmanager';

export class MyStack extends cdk.Stack {
  constructor(scope: cdk.App, id: string, props?: cdk.StackProps) {
    super(scope, id, props);

    const dbPasswordArn = this.node.tryGetContext('dbPassword');
    const apiKeyArn = this.node.tryGetContext('apiKey');

    if (!dbPasswordArn || !apiKeyArn) {
      throw new Error('Missing required context variables: dbPassword and apiKey');
    }

    const dbPasswordSecret = secretsmanager.Secret.fromSecretCompleteArn(this, 'DbPasswordSecret', dbPasswordArn);
    const apiKeySecret = secretsmanager.Secret.fromSecretCompleteArn(this, 'ApiKeySecret', apiKeyArn);

    const myFunction = new lambda.Function(this, 'MyFunction', {
      runtime: lambda.Runtime.NODEJS_18_X,
      handler: 'index.handler',
      code: lambda.Code.fromAsset('lambda'),
      environment: {
        DATABASE_PASSWORD: dbPasswordSecret.secretValueFromJson('password').toString(), // Adjust key as needed
        API_KEY: apiKeySecret.secretValueFromJson('key').toString(), // Adjust key as needed
      },
    });

    dbPasswordSecret.grantRead(myFunction);
    apiKeySecret.grantRead(myFunction);
  }
}
```

### 4.3 Attack Vector Exploration

1.  **Compromised Source Code Repository:**
    *   **Scenario 1: Public Repository:**  A developer accidentally pushes CDK code containing hardcoded secrets to a public GitHub repository.  Automated scanners (or malicious actors) quickly identify and extract the secrets.
    *   **Scenario 2: Private Repository Breach:**  An attacker gains unauthorized access to a private repository (e.g., through a phishing attack targeting a developer, a compromised CI/CD pipeline, or a vulnerability in the repository hosting service).
    *   **Scenario 3: Insider Threat:**  A disgruntled employee with access to the private repository copies the code and extracts the secrets.

2.  **Exploitation:** Once the attacker has the secrets, they can:
    *   **Directly Access Resources:**  Use the secrets to authenticate to databases, APIs, or other services.
    *   **Data Exfiltration:**  Steal sensitive data from databases.
    *   **Service Disruption:**  Modify or delete resources, causing downtime.
    *   **Lateral Movement:**  Use the compromised resources as a stepping stone to attack other parts of the infrastructure.
