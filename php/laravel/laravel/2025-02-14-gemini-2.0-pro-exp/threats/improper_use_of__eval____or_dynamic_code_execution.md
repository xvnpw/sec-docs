## 2. Deep Analysis of the Threat

### 2.1. Threat Mechanics

The core of this threat lies in allowing an attacker to inject arbitrary PHP code into the application's execution flow.  This is achieved by providing malicious input that is then used *without proper sanitization or validation* in a context where dynamic code execution is performed.

**Example (Vulnerable Code - `eval()`):**

```php
// In a controller (extremely dangerous and unrealistic, but illustrative)
public function executeCode(Request $request) {
    $code = $request->input('code');
    eval($code); // NEVER DO THIS
    return "Code executed.";
}
```

An attacker could send a POST request with `code=phpinfo();` to reveal server information, or much worse: `code=system('rm -rf /');` (on a poorly configured system, this could delete the entire filesystem).

**Example (Vulnerable Code - Dynamic Class Instantiation):**

```php
// In a controller
public function processAction(Request $request) {
    $actionClass = $request->input('action'); // e.g., "UserAction", "AdminAction"
    $action = new $actionClass(); // DANGEROUS if $actionClass is not validated
    $action->execute();
    return "Action processed.";
}
```
