# Deep Analysis: Misconfigured HTTP/2 Attack Surface in Helidon Applications

## 1. Objective, Scope, and Methodology

### 1.1 Objective

The objective of this deep analysis is to thoroughly examine the "Misconfigured HTTP/2" attack surface within applications built using the Helidon framework.  This includes understanding the specific vulnerabilities that can arise from misconfiguration, identifying the Helidon components involved, detailing exploitation techniques, and providing concrete, actionable mitigation strategies for both developers and operators.  The ultimate goal is to provide the development team with the knowledge necessary to prevent and mitigate this class of attacks.

### 1.2 Scope

This analysis focuses exclusively on HTTP/2 misconfigurations *within the Helidon framework itself*.  It does *not* cover:

*   General HTTP/2 vulnerabilities that are independent of Helidon (e.g., vulnerabilities in underlying network libraries, unless Helidon exposes a misconfiguration of those).
*   Misconfigurations of reverse proxies or load balancers *in front of* the Helidon application (although these can exacerbate the impact, the root cause we're analyzing is within Helidon).
*   Other attack surfaces within the Helidon application (e.g., SQL injection, XSS).

The scope is limited to Helidon's configuration and handling of HTTP/2, as exposed through its APIs, configuration files (e.g., `application.yaml`, `microprofile-config.properties`), and programmatic configuration options.  We will consider both Helidon SE and Helidon MP.

### 1.3 Methodology

The analysis will follow these steps:

1.  **Helidon Component Identification:** Identify the specific Helidon modules and classes responsible for HTTP/2 processing.  This will involve reviewing the Helidon source code and documentation.
2.  **Configuration Parameter Analysis:**  Examine all configurable parameters related to HTTP/2 within Helidon.  This includes identifying default values, allowed ranges, and the security implications of different settings.
3.  **Exploitation Scenario Detailing:**  Expand on the provided "HPACK bombing" example and explore other potential HTTP/2-based attacks that could exploit Helidon misconfigurations.  This will include researching known HTTP/2 vulnerabilities and adapting them to the Helidon context.
4.  **Mitigation Strategy Refinement:**  Provide specific, actionable recommendations for developers and operators, including code examples, configuration snippets, and monitoring strategies.  These recommendations will be tailored to Helidon's specific features.
5.  **Testing Recommendations:** Suggest methods for testing the application's resilience to HTTP/2-based attacks.

### 2.1 Helidon Component Identification

The core Helidon components relevant to HTTP/2 configuration and handling are:

*   **`io.helidon.webserver` (Helidon SE):**  This module provides the core web server functionality in Helidon SE.  It includes classes for configuring and managing HTTP/2 connections.  Key classes include `WebServer`, `Routing`, and related configuration classes.
*   **`io.helidon.webserver.http2` (Helidon SE):** Specifically handles HTTP/2 within the webserver.  Classes like `Http2Connection`, `Http2Stream`, and configuration options within `Http2Config` are crucial.
*   **Helidon MP's Web Server Integration:** Helidon MP builds upon Helidon SE's web server.  While the underlying HTTP/2 handling is still managed by the SE components, MP provides higher-level abstractions and configuration mechanisms (e.g., through MicroProfile Config).
*   **Netty:** Helidon uses Netty as its underlying network framework. While we are focusing on Helidon's configuration, it's important to understand that Helidon's HTTP/2 support is ultimately implemented using Netty's HTTP/2 capabilities.  Misconfigurations in Helidon can expose underlying Netty vulnerabilities.

### 2.3 Exploitation Scenario Detailing

Beyond the HPACK bombing example, several other HTTP/2 misconfigurations can lead to vulnerabilities:

*   **Stream Multiplexing Abuse (Excessive Concurrent Streams):**  HTTP/2 allows multiple streams to be multiplexed over a single connection.  Helidon, by default, might have a high or unlimited number of concurrent streams allowed.  An attacker could open a large number of streams, each making small requests, without closing them.  This can exhaust server resources (memory, CPU, file descriptors) even if individual requests are small and seemingly harmless.  This is different from a traditional HTTP/1.1 DoS because the attacker doesn't need many connections; a single connection can be leveraged for many parallel requests.

*   **SETTINGS Flood:**  The attacker sends a continuous stream of `SETTINGS` frames.  Each `SETTINGS` frame requires the server to acknowledge it.  While Helidon/Netty likely has some protection against this, a misconfiguration (e.g., excessively large buffers or long timeouts for acknowledging) could allow an attacker to consume resources by forcing the server to process and respond to a flood of these frames.

*   **PING Flood:** Similar to the SETTINGS flood, but using PING frames.  The server must respond with a PONG frame for each PING.  Again, Helidon/Netty likely has built-in protections, but misconfiguration could weaken them.

*   **Header Manipulation (Invalid/Malformed Headers):**  HTTP/2 uses HPACK for header compression.  An attacker could send crafted, invalid, or overly complex headers that trigger errors or unexpected behavior in Helidon's HPACK implementation.  This could lead to crashes, resource exhaustion, or potentially even code execution if a buffer overflow or similar vulnerability exists in the parsing logic.  This is less about configuration and more about Helidon's robustness, but configuration *can* influence the size limits and complexity allowed.

*   **Stream Priority Manipulation:** HTTP/2 allows clients to specify stream priorities.  An attacker could manipulate these priorities to starve lower-priority streams, potentially impacting legitimate users or critical application functions.  Helidon's configuration might allow for fine-grained control over how priorities are handled, and a misconfiguration could make this attack more effective.

*   **RST_STREAM Flooding:**  An attacker rapidly opens and closes streams using `RST_STREAM` frames.  This can consume resources related to stream creation and teardown, even if the streams themselves don't transmit much data.  Helidon's configuration might allow limits on the rate of stream creation/destruction.

* **CONTINUATION Flood:** HTTP headers that are too large to fit in a single frame are split across multiple frames using CONTINUATION frames. An attacker can send a HEADERS frame followed by a large number of empty CONTINUATION frames. This can consume resources on the server as it has to process each frame.

### 2.4 Mitigation Strategy Refinement

Here are specific, actionable mitigation strategies, with examples where possible:

**2.4.1 Developer Mitigations (Helidon Configuration)**

*   **Limit Header Size:**  Use `Http2Config` (Helidon SE) or equivalent MicroProfile Config properties (Helidon MP) to set strict limits on HTTP/2 header sizes.  This is the *primary* defense against HPACK bombing.

    ```java
    // Helidon SE Example (programmatic configuration)
    Http2Config http2Config = Http2Config.builder()
            .maxHeaderListSize(8192) // Limit header list size to 8KB (adjust as needed)
            .initialWindowSize(65535) // Limit initial window size
            .maxFrameSize(16384) // Limit maximum frame size
            .build();

    WebServer webServer = WebServer.builder()
            .routing(routing)
            .addProtocol(Http2Config.PROTOCOL, http2Config) // Apply the configuration
            .build();
    ```

    ```yaml
    # Helidon SE Example (application.yaml)
    server:
      port: 8080
      http2:
        max-header-list-size: 8192
        initial-window-size: 65535
        max-frame-size: 16384
    ```

    ```properties
    # Helidon MP Example (microprofile-config.properties)
    mp.http.server.http2.max-header-list-size=8192
    mp.http.server.http2.initial-window-size=65535
    mp.http.server.http2.max-frame-size=16384
    ```

*   **Limit Concurrent Streams:**  Set a reasonable limit on the maximum number of concurrent streams per connection.  This prevents stream multiplexing abuse.

    ```java
    // Helidon SE Example
    Http2Config http2Config = Http2Config.builder()
            .maxConcurrentStreams(100) // Limit to 100 concurrent streams
            .build();
    ```

    ```yaml
    # Helidon SE Example (application.yaml)
    server:
      http2:
        max-concurrent-streams: 100
    ```

    ```properties
    # Helidon MP Example (microprofile-config.properties)
    mp.http.server.http2.max-concurrent-streams=100
    ```

*   **Enable and Configure Flow Control:**  HTTP/2 flow control is crucial.  Ensure it's enabled (it should be by default) and that the window sizes are appropriately configured.  Too large a window size can allow an attacker to send excessive data before being throttled.

*   **Review and Harden Netty Configuration (if exposed):** If Helidon allows direct configuration of underlying Netty parameters, review those related to HTTP/2 for security best practices.  This is generally *not* recommended unless you have a deep understanding of Netty.

*   **Use Helidon's Built-in Security Features:** Helidon provides security features like request filtering and authentication.  While these don't directly mitigate HTTP/2 protocol-level attacks, they can limit the impact by restricting access to vulnerable endpoints.

*   **Stay Updated:** Regularly update Helidon to the latest version to benefit from security patches and improvements in HTTP/2 handling.

**2.4.2 Operator Mitigations (Monitoring and Infrastructure)**

*   **Monitor Resource Usage:**  Use monitoring tools (e.g., Prometheus, Grafana, Helidon's built-in metrics) to track:
    *   CPU usage
    *   Memory usage
    *   Number of open connections
    *   Number of active HTTP/2 streams
    *   HTTP/2 error rates (e.g., `RST_STREAM` counts)
    *   Request latency and throughput

    Sudden spikes or sustained high values in these metrics could indicate an attack.

*   **Implement Rate Limiting (at the network level):**  While the primary mitigation is within Helidon, a Web Application Firewall (WAF) or API gateway in front of your Helidon application can provide an additional layer of defense by rate-limiting connections or requests.  This can help mitigate some flooding attacks.  However, be aware that rate limiting based solely on IP address can be bypassed by attackers using distributed attacks.

*   **Alerting:** Configure alerts based on the monitored metrics.  For example, trigger an alert if the number of concurrent HTTP/2 streams exceeds a predefined threshold or if the CPU usage remains consistently high.

*   **Incident Response Plan:** Have a plan in place to respond to potential HTTP/2-based attacks.  This might involve temporarily disabling HTTP/2, blocking specific IP addresses, or scaling up resources.

### 2.5 Testing Recommendations

*   **Fuzz Testing:** Use a fuzzing tool specifically designed for HTTP/2 (e.g., h2spec, `nghttp2`'s `h2load` with crafted inputs) to send malformed or unexpected HTTP/2 frames to your Helidon application.  This can help identify vulnerabilities in Helidon's parsing and handling logic.

*   **Load Testing with Malicious Traffic:**  Simulate realistic attack scenarios (e.g., high concurrent streams, large headers) using load testing tools.  Monitor the application's behavior and resource usage under stress.

*   **Penetration Testing:**  Engage a security professional to conduct penetration testing, specifically targeting the HTTP/2 implementation.

*   **Static Analysis:** Use static analysis tools to scan the Helidon codebase and your application code for potential vulnerabilities related to HTTP/2 handling.

* **Unit and Integration Tests:** Write unit and integration tests that specifically target the HTTP/2 configuration and handling within your application
