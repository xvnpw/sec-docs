## Spring Boot Actuator

Spring Boot Actuator is a sub-project of the Spring Boot framework that provides production-ready features to help you monitor and manage your application. It offers various endpoints to check the health, metrics, and other information about the running application. These endpoints are particularly useful for monitoring and managing the application in a production environment.

### 1. **What it is:**

-   **Definition:** Spring Boot Actuator is a set of tools that offer production-ready features to help you monitor and manage your application when it's pushed to production.
-   **Purpose:** It helps in monitoring and managing the application by exposing various endpoints.
-   **Endpoints:** These endpoints allow you to interact with and monitor your application, check its health, metrics, environment, and more.

### 2. **Default Endpoints:**

By default, Spring Boot exposes two main endpoints over HTTP:

-   `/health`: Shows application health information.
-   `/info`: Displays arbitrary application info.

However, many other endpoints can be enabled, each providing different insights into the application.

### 3. **Security Concerns:**

-   **Exposure:** The main issue with Actuator endpoints is that they can expose sensitive information about your application if not properly secured.
-   **Risk:** Unauthorized access to these endpoints can lead to information disclosure, denial of service, and even remote code execution in some scenarios.

### 4. **Enabling and Disabling Endpoints:**

-   **Enabling:** Endpoints can be enabled individually using `management.endpoint.<id>.enabled` property.
-   **Disabling:** Similarly, they can be disabled using the same property set to `false`.
-   **Example:** To enable the `shutdown` endpoint, you would set `management.endpoint.shutdown.enabled=true` in your `application.properties` or `application.yml` file.

### 5. **Web Exposure:**

-   **Configuration:** The exposure of endpoints over HTTP is controlled by `management.endpoints.web.exposure.include` and `management.endpoints.web.exposure.exclude`.
-   **Default:** By default, only `health` and `info` are exposed over the web.
-   **Customization:** You can customize which endpoints are exposed. For instance, to expose all endpoints, you can set `management.endpoints.web.exposure.include=*`.

### 6. **Security:**

-   **Importance:** It's crucial to secure Actuator endpoints, especially if they are exposed over the web.
-   **Spring Security:** If Spring Security is on the classpath, endpoints are secured by default using Spring Security's content negotiation strategy.
-   **Custom Security:** You can define custom security rules for Actuator endpoints.

### 4.2. Information Disclosure Risks

Exposed Actuator endpoints can leak a variety of sensitive information, including:

*   **`/actuator/env`:**  This is one of the most dangerous endpoints if exposed. It reveals *all* environment variables, including:
    *   Database credentials (usernames, passwords, URLs)
    *   API keys for third-party services
    *   Cloud provider credentials (AWS access keys, Azure service principals, etc.)
    *   Internal network configurations
    *   Application configuration properties (potentially including secrets)
    *   Java system properties (including `JAVA_OPTS`, which might contain sensitive flags)

*   **`/actuator/configprops`:**  Displays all `@ConfigurationProperties` beans.  While not as immediately dangerous as `/env`, this can reveal sensitive configuration details, especially if best practices for separating secrets from configuration are not followed.  It can expose database URLs, internal service endpoints, and other configuration parameters.

*   **`/actuator/beans`:**  Shows all Spring beans and their dependencies.  This can reveal the internal architecture of the application, including class names, package structures, and dependencies.  This information can be used by an attacker to understand the application's inner workings and identify potential attack vectors.

*   **`/actuator/mappings`:**  Lists all request mappings (e.g., `@RequestMapping`, `@GetMapping`, `@PostMapping`).  This exposes the application's API surface, making it easier for an attacker to find potential vulnerabilities in other parts of the application.

*   **`/actuator/httptrace` (or `/actuator/httpexchanges` in newer Spring Boot versions):**  Provides a trace of recent HTTP requests and responses.  This can include sensitive data such as:
    *   Session IDs (if not properly handled)
    *   Authentication tokens (if included in headers or request bodies)
    *   User-submitted data (including potentially sensitive form data)
    *   Internal IP addresses and hostnames

*   **`/actuator/threaddump`:**  Provides a thread dump of the JVM.  While primarily useful for debugging, it can also reveal information about running processes and potentially sensitive data held in memory.

*   **`/actuator/heapdump`:**  Creates a heap dump file (HPROF format).  This is *extremely* dangerous if exposed, as it contains a snapshot of the entire JVM heap, including all objects and their data.  An attacker could analyze this file offline to extract sensitive information.

*   **`/actuator/loggers`:**  Allows viewing and *modifying* the logging configuration at runtime.  An attacker could disable logging to cover their tracks or increase logging verbosity to capture sensitive information.

*   **`/actuator/metrics`:**  Exposes application metrics.  While not inherently sensitive, this can provide information about application load, performance, and resource usage, which could be used in a denial-of-service attack or to identify weaknesses.

*   **`/actuator/caches`:** Exposes information about the application's caches. An attacker could potentially clear caches, impacting performance.

### 4.3. Denial of Service (DoS) Risks

*   **`/actuator/shutdown`:**  If enabled and exposed, this endpoint allows anyone to gracefully shut down the application.  This is a direct and obvious DoS vector.

*   **`/actuator/loggers`:**  An attacker could set the logging level to `TRACE` for all loggers, potentially causing excessive disk I/O and performance degradation.

*   **`/actuator/heapdump`:** Generating a heap dump can be a resource-intensive operation, especially for large applications.  Repeated requests to this endpoint could lead to a denial of service.

*   **Resource Exhaustion:** Even seemingly harmless endpoints like `/actuator/metrics` or `/actuator/beans` could be abused to cause resource exhaustion if an attacker makes a large number of requests in a short period.

### 4.4. Remote Code Execution (RCE) Risks

While Actuator itself doesn't directly provide an RCE endpoint, it can be a crucial component in a multi-stage attack:

*   **Information Gathering:**  Endpoints like `/actuator/env`, `/actuator/configprops`, and `/actuator/beans` can provide an attacker with the information needed to craft a targeted exploit against other vulnerabilities in the application or its dependencies. For example, knowing the exact version of a vulnerable library used by the application (revealed through `/actuator/beans` or environment variables) is critical for exploiting known CVEs.

*   **Configuration Manipulation:**  If combined with another vulnerability (e.g., a deserialization vulnerability or a misconfigured Spring Cloud Config Server), an attacker might be able to use `/actuator/env` or `/actuator/refresh` to inject malicious configuration properties that lead to RCE.  This is a more complex attack scenario, but it has been demonstrated in the past.

*   **Log4Shell-like Vulnerabilities:**  The `/actuator/env` endpoint was a key component in exploiting the Log4Shell vulnerability (CVE-2021-44228).  Attackers could use it to inject malicious JNDI lookup strings into environment variables, which were then logged by Log4j, triggering the vulnerability.  This highlights the danger of exposing environment variables.

### 4.5. Mitigation Strategies: Detailed Breakdown

Here's a detailed breakdown of the mitigation strategies, with code examples and best practices:

**4.5.1. Restrict Access with Spring Security:**

This is the *most important* mitigation.  Spring Security should be used to require authentication and authorization for all Actuator endpoints.

*   **Basic Configuration (Java Config):**

```java
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.config.annotation.web.configuration.EnableWebSecurity;
import org.springframework.security.core.userdetails.User;
import org.springframework.security.core.userdetails.UserDetails;
import org.springframework.security.provisioning.InMemoryUserDetailsManager;
import org.springframework.security.web.SecurityFilterChain;
import org.springframework.security.web.util.matcher.AntPathRequestMatcher;

@Configuration
@EnableWebSecurity
public class ActuatorSecurityConfig {

    @Bean
    public InMemoryUserDetailsManager userDetailsService() {
        UserDetails user = User.withDefaultPasswordEncoder()
                .username("admin")
                .password("adminPassword") // Change this in production!
                .roles("ACTUATOR_ADMIN")
                .build();
        return new InMemoryUserDetailsManager(user);
    }

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests((authz) -> authz
                .requestMatchers(new AntPathRequestMatcher("/actuator/**")).hasRole("ACTUATOR_ADMIN")
                .anyRequest().authenticated() // Secure other endpoints as needed
            )
            .httpBasic(); // Use basic authentication (for simplicity in this example)
        return http.build();
    }
}
```

*   **Explanation:**
    *   `@EnableWebSecurity`: Enables Spring Security.
    *   `InMemoryUserDetailsManager`:  Creates an in-memory user store (for demonstration purposes; use a proper user store in production).  **Important:**  Change the default password!
    *   `SecurityFilterChain`: Defines the security rules.
        *   `requestMatchers(new AntPathRequestMatcher("/actuator/**")).hasRole("ACTUATOR_ADMIN")`:  This is the crucial part.  It requires the `ACTUATOR_ADMIN` role for *any* request to `/actuator` or its sub-paths.
        *   `.anyRequest().authenticated()`:  Requires authentication for all other requests (you'll likely want more specific rules here).
        *   `.httpBasic()`:  Enables basic authentication.  You could use other authentication methods (e.g., form login, OAuth2) as well.

*   **Best Practices:**
    *   **Use a strong password:**  Never use default or easily guessable passwords.
    *   **Use a proper user store:**  In production, use a database, LDAP, or another robust user store instead of `InMemoryUserDetailsManager`.
    *   **Least Privilege:**  Grant the `ACTUATOR_ADMIN` role (or a similarly named custom role) only to users who absolutely need access to Actuator endpoints.
    *   **Consider different roles:**  You might want different roles for different endpoints (e.g., `ACTUATOR_READ` for read-only access, `ACTUATOR_WRITE` for endpoints that modify state).
    *   **CSRF Protection:** While less critical for Actuator endpoints (which are typically accessed by monitoring tools, not browsers), consider enabling CSRF protection if you have endpoints that modify state. Spring Security enables CSRF protection by default.
