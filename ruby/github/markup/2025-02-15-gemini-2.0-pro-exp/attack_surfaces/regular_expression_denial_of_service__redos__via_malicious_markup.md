# Regular Expression Denial of Service (ReDoS) Explained

## 1. What is ReDoS? (The Problem)

ReDoS, or Regular Expression Denial of Service, is a type of cyberattack that exploits a vulnerability in how some regular expression engines work.  It causes a seemingly harmless regular expression to take an extremely long time (potentially *years*) to process certain specially crafted inputs, effectively freezing or crashing the application.

**Analogy:** Imagine a maze with many branching paths. A well-designed maze has clear routes, and you can find your way through it relatively quickly.  A poorly designed maze, however, might have loops and nested paths that, if you take a wrong turn, could trap you for hours, forcing you to backtrack through countless combinations.  ReDoS is like feeding a regular expression engine a "poorly designed maze" in the form of a malicious input string. The engine gets stuck in a computational loop, trying every possible (and often exponentially increasing) combination before finally giving up.

## 2. How does it work? (The Mechanism)

ReDoS exploits a phenomenon called "catastrophic backtracking."  This happens when a regular expression engine uses backtracking to explore many different possible matches within a string, and the number of possibilities grows exponentially with the input length.

**Vulnerable Regex Example (Very Simple):**

```regex
(a+)+$
```

**Explanation:**

*   `a+`:  Matches one or more "a" characters.
*   `( ... )+`:  The outer `+` means the entire group `(a+)` can be repeated one or more times.
*   `$`: Matches the end of the string.

**Why it's vulnerable:**

Consider the input string "aaaaaaaaaaaaaaaaaaaaaaaaaaaa!".  The engine first tries to match as many "a"s as possible with the inner `a+`.  Then, the outer `+` tries to repeat this match.  Because of the nested quantifiers (`+` inside `+`), the engine explores many *redundant* combinations:

1.  `(aaaaaaaaaaaaaaaaaaaaaaaaaaaa)` (one big group)
2.  `(aaaaaaaaaaaaaaaaaaaaaaaaaaa)(a)` (two groups)
3.  `(aaaaaaaaaaaaaaaaaaaaaaaaaa)(aa)`
4.  `(aaaaaaaaaaaaaaaaaaaaaaaaaa)(a)(a)`
5.  ...and so on...

Each of these combinations fails because of the `!` at the end (which doesn't match `a`).  The engine *backtracks* and tries *another* combination.  The number of combinations grows exponentially with the number of "a"s.  A slightly longer string can cause the processing time to increase dramatically.  This is the "catastrophic backtracking."  The `$` anchor, while seemingly simple, forces the engine to check *all* these combinations before failing. Without the `$`, it might find a partial match earlier and avoid some backtracking.

**More complex, realistic examples often involve:**

*   **Nested quantifiers:**  Like `(a+)+`, `(a*)*`, `(a|b+)+`.
*   **Overlapping alternations:** Like `(a|aa)+`, where the alternatives can match the same input in different ways.
*   **Lookarounds (less common, but possible):**  Poorly constructed lookarounds can also contribute to backtracking issues.

## 3. Impact of ReDoS

A successful ReDoS attack can have the following impacts:

*   **Application Unavailability:** The application thread handling the regular expression becomes unresponsive, unable to process other requests.  This effectively makes the application unavailable to users.
*   **Server Resource Exhaustion:** The affected thread consumes 100% of a CPU core.  If multiple requests trigger the ReDoS, multiple cores can be maxed out, leading to overall system slowdown or even a complete crash.
*   **Denial of Service:**  The primary goal of the attacker is achieved: legitimate users are denied access to the service.
*   **Potential for Horizontal Impact:** If the vulnerable regex processing is part of a shared service or library, other applications using that service could also be affected.
