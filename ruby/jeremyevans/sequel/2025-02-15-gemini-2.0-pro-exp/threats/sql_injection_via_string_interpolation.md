### 4.1 Vulnerability Mechanics

The core issue is that string interpolation in Ruby happens *before* the resulting string is passed to Sequel.  Sequel's query building and parameterization mechanisms operate on the *final* string.  If that string contains user-supplied data directly embedded via interpolation, Sequel *cannot* distinguish between the intended SQL code and the attacker's malicious input.

**Example (Vulnerable):**

```ruby
username = params[:username] # Assume this comes from a web form
DB.fetch("SELECT * FROM users WHERE username = '#{username}'")
```

If `params[:username]` is `admin' --`, the resulting SQL becomes:

```sql
SELECT * FROM users WHERE username = 'admin' --'
```

The `--` comments out the rest of the query, effectively bypassing any password check.  The attacker has injected SQL.  Even though this *looks* like a Sequel query, the interpolation happens *outside* of Sequel's control.

**Another Vulnerable Example (Subtle):**

```ruby
column_name = params[:column] # User controls which column to filter on
value = params[:value]
DB[:users].where("#{column_name} = ?", value)
```

Even though a placeholder (`?`) is used for the *value*, the *column name* is interpolated.  An attacker could set `column_name` to `name; DROP TABLE users; --`, resulting in:

```sql
SELECT * FROM users WHERE name; DROP TABLE users; -- = ?
```

This demonstrates that *any* string interpolation within the SQL fragment passed to Sequel is dangerous, even if *some* parts are parameterized.
