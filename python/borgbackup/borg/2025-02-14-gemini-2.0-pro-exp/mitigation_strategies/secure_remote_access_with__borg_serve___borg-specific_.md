## 3. Technical Deep-Dive

Let's examine how each of the `borg serve` options works under the hood:

*   **`--read-only`:** This option fundamentally alters the way Borg handles repository access at a low level.  When `borg serve` is invoked with `--read-only`, it sets a flag within the Borg server process that prevents any write operations to the repository.  This is enforced by the `borg.repository.Repository` class, which checks this flag before executing any methods that would modify the repository (e.g., `create`, `delete`, `prune`, `compact`, etc.).  If the flag is set, these methods will raise a `ReadOnlyError`.  This is a very robust mechanism because it's enforced at the repository object level, not just at the command-line argument parsing level.  It's not simply a superficial check; it's deeply integrated into the core logic of how Borg interacts with the repository.

*   **`--restrict-to-repository <path>`:** This option limits the client's view to a single, specified repository.  The server process, upon receiving a connection, checks the requested repository path against the allowed path.  If they don't match, the connection is immediately closed.  This is handled by the `borg.remote.serve.Server` class.  The server maintains a list of allowed repositories (in this case, just one) and compares the client's request against this list.  Crucially, this check happens *before* any repository operations are performed, preventing even listing of other repositories.

*   **`--restrict-to-path <path>`:** This is the most granular control.  It allows access *only* to a specific path *within* a repository.  This is implemented by extending the repository access checks to include path validation.  The `borg.remote.serve.Server` class, after verifying the repository, further checks if the requested path is a subpath of the allowed path.  This is a string-based comparison, but Borg handles path normalization to prevent bypasses using tricks like `../` or symbolic links (within the repository).  The server effectively "chroots" the client's view to the specified path.  Importantly, this restriction applies to *all* Borg operations, including listing archives, extracting files, etc.

*   **Error Handling:** Borg's error handling is generally robust.  If a client attempts an unauthorized operation (writing to a read-only repository, accessing a restricted path), Borg returns a clear error message indicating the reason for the failure.  These errors are logged on the server side, providing an audit trail.  The client receives a non-zero exit code, allowing for scripting and automation to detect failures.

*   **Edge Cases:**
    *   **Symbolic Links (within the repository):** Borg handles symbolic links *within* the repository correctly, respecting the `--restrict-to-path` limitations.  A client cannot use a symlink to escape the restricted path.  However, Borg does *not* follow symbolic links *outside* the repository (which is a good security practice).
    *   **Hard Links (within the repository):** Hard links are treated as distinct files.  If a hard link points to a file outside the restricted path, the client will not be able to access it through the hard link.
    *   **Empty `--restrict-to-path`:**  An empty `--restrict-to-path` (e.g., `--restrict-to-path ""`) effectively denies access to the entire repository, as no path will be a subpath of the empty string. This is a useful way to effectively disable access without shutting down the server.
    *   **Path Traversal Attacks:** Borg's path normalization prevents common path traversal attacks (e.g., using `../` to escape the restricted directory).
    *   **Race Conditions:** Borg uses appropriate locking mechanisms to prevent race conditions that could potentially allow unauthorized access during concurrent operations. This is handled at the repository level.

## 4. Implementation Review

Based on the "Currently Implemented" and "Missing Implementation" sections, our current setup is highly vulnerable.  Here's a breakdown:

*   **Missing `--read-only`:**  This is a critical omission.  Any client with network access and the correct credentials (even if intended for read-only restores) can *potentially* modify or delete the repository.  This includes accidental modifications, malicious actions by a compromised client, or even a misconfigured client script.
*   **Missing `--restrict-to-repository`:** This allows a client to potentially list and access *all* repositories served by the Borg instance.  This is a significant information disclosure vulnerability and increases the attack surface.  A client could discover the names and potentially the contents of other repositories, even if they don't have the keys to decrypt them.
*   **Missing `--restrict-to-path`:**  This further exacerbates the access control problem.  Even within a given repository, a client has full access to all archives and files.  This violates the principle of least privilege.

The lack of these restrictions means we are relying entirely on network-level security and client-side trust.  This is insufficient.
