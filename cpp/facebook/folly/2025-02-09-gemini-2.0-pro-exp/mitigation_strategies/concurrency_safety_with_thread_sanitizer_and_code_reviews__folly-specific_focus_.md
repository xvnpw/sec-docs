Developers must understand that continuations (callbacks attached to futures) *may not run on the same thread* that created the future or completed it.  This is a common source of data races if shared mutable state is accessed within continuations without proper synchronization.  The strategy needs to explicitly address this.
    *   **`via()`:** The strategy mentions using higher-level abstractions like `folly::via`.  This is excellent.  `via()` allows specifying the executor on which a continuation should run, providing more control over threading and reducing the risk of unexpected thread switches.  Code reviews should encourage its use.

    *   **Ownership and Borrowing:**  If shared mutable state is unavoidable, the strategy should promote clear ownership and borrowing patterns to minimize the risk of data races.  Rust's ownership model, while not directly applicable to C++, provides a good conceptual framework.

    *   **`folly::collect`:**  This provides higher-level operations on collections, often avoiding the need for explicit concurrency management.  Code reviews should encourage its use where appropriate.
    *   **`folly::coroutine` (if applicable):** If the project uses C++20 coroutines, Folly provides integration that can simplify asynchronous programming and reduce the risk of errors.

*   **Avoid blocking operations in folly thread pools:**
    * **Identify blocking operations:** Code review should identify any blocking operations, such as I/O operations, locks, or long computations.
    * **Use asynchronous alternatives:** Encourage the use of asynchronous alternatives, such as folly::futures, folly::coroutine, or non-blocking I/O.
    * **Offload blocking operations:** If blocking operations are unavoidable, offload them to a separate thread pool dedicated to blocking tasks.

### 5. Best Practice Comparison

The strategy aligns with many best practices for concurrent programming:

*   **Use of a Thread Sanitizer:**  TSan is a widely recognized and effective tool for detecting data races.
*   **Emphasis on Code Reviews:**  Code reviews are essential for catching concurrency bugs that might be missed by automated tools.
*   **Minimizing Shared Mutable State:**  This is a fundamental principle of safe concurrent programming.
*   **Using Higher-Level Abstractions:**  This reduces the complexity of concurrency management and the likelihood of errors.

However, it falls short in the *specificity* and *formalization* of the code review process and the lack of developer training.
